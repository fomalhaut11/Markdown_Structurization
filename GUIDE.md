# 给 AI 一份项目心智模型：让编码助手真正"懂"你的项目

> 这是一套轻量级的项目文档模板，用最少的维护成本，让 AI 编码助手从"勉强能用"变成"真正好用"。

---

## 一、问题：AI 编码助手为什么在真实项目中"不好用"

以下三个场景在实际开发中非常常见：

**场景 1：不理解模块边界**
让 AI 修改用户认证逻辑，结果它把 session 管理的代码也一并重写了——因为它不知道 auth 和 session 是两个独立模块，各有明确的职责边界。开发者不得不花时间还原被误改的代码。

**场景 2：不了解代码风格**
项目统一使用 Repository 模式做数据访问，AI 却在新功能中直接写了裸 SQL 查询。代码能运行，但 review 时只能全部打回——因为它完全不知道团队有这个约定。

**场景 3：不清楚跨模块影响**
让 AI 修改了订单模块的一个接口返回值，改完了，测试也通过了。但上线后库存模块报错——因为库存模块依赖该接口的旧格式，而 AI 对这层依赖关系一无所知。

这三个场景指向同一个根因：**LLM 缺少项目的心智模型**。

它不知道项目有哪些模块、模块之间如何通信、团队有什么约定。每次对话，它都相当于一个"刚入职、没有上下文、只能看到眼前代码"的新成员。

---

## 二、核心思路：给 AI 一份"项目心智模型"

类比一下——一个新人入职时，通常会经历这样的流程：先了解项目整体架构，再熟悉负责模块的接口和约定，最后明确提交代码前的规范要求。有了这些上下文，新人就能独立工作。

AI 编码助手也是同样的逻辑。它不需要一本 500 页的文档，它需要的是：
- **架构全景**：项目的整体结构和模块划分
- **接口契约**：每个模块对外暴露什么、依赖什么
- **工作流程**：接到任务后按什么步骤执行

这套模板做的就是这件事——用最少的文档（通常不超过 10 个文件），给 AI 一个结构化的项目心智模型。

## 三、三层信息架构

模板把信息分成三层，每层解决不同的问题：

```
┌─────────────────────────────────────────────┐
│            CLAUDE.md（常驻层）                │
│                                             │
│  每次对话自动加载 · 控制在 20 行以内          │
│  只放两样东西：绝对约束 + 知识库入口          │
├─────────────────────────────────────────────┤
│            docs/（知识层）                    │
│                                             │
│  按需加载 · AI 需要时主动查阅                │
│  architecture.md   → 项目全景               │
│  modules/*.md      → 模块接口契约            │
│  data-flows.md     → 跨模块数据流            │
├─────────────────────────────────────────────┤
│         .claude/skills/（行为层���             │
│                                             │
│  按任务类型触发 · 固化工作流程                │
│  understand-project / add-feature /          │
│  fix-bug / review-code / update-docs         │
└─────────────────────────────────────────────┘
```

为什么分三层？因为上下文窗口（token）是有限资源。

- **常驻层**每次对话都会消耗 token，因此必须极度精简。20 行约 500 token，对 200K 上下文窗口几乎无感。但如果将 200 行文档放入常驻层，每次对话都在浪费上下文空间。
- **知识层**按需加载。修改 auth 模块时只读 auth 的文档，无需将所有模块文档一次性加载。
- **行为层**在触发具体任务时才生效，定义标准化的执行流程。

一句话总结：**该常驻的精简常驻，该按需的按需加载，该触发的按需触发。**

## 四、四个设计原则

### 1. 约束前置：预防 > 纠正

传统做法是让 AI 先生成代码，人再 review 纠正。
这套模板的思路是：**把约束前置到 AI 动手之前**。

例如 CLAUDE.md 中写明"修改模块前必须先读该模块文档"，AI 在修改代码之前就会主动读取接口契约，生成的代码天然符合约定。预防一个错误的成本，远低于 review 时发现再打回重写。

### 2. 渐进式披露：按需加载节省 token

并非一次性将所有文档加载给 AI，而是让它按需取用：

| 场景 | AI 需要的上下文 |
|------|----------------|
| 修复一个拼写错误 | CLAUDE.md（20 行） |
| 改 auth 模块的 bug | + modules/auth.md |
| 追踪订单流程的问题 | + data-flows.md 的"下单流程" |
| 第一次了解项目 | + architecture.md + 所有模块文档 |

日常小改动只需要很少的上下文，AI 可以将更多 token 用于思考和代码生成。

### 3. 文档即契约：接口文档不是说明书

模块文档的核心不是"解释这个模块做什么"，而是**定义它对外的契约**：

- 公开接口的签名、参数、返回值、副作用
- 依赖方向：谁可以调用谁，谁不能调用谁
- 错误处理约定

这样当 AI 修改某个模块时，不仅知道内部如何改动，还清楚改动会影响哪些下游模块。回到前面的场景 3——如果订单模块文档中明确标注了"库存模块依赖此接口"，AI 就不会随意修改返回格式。

### 4. 闭环维护：skill 流程强制更新文档

文档最怕的就是过时。这套模板的解决方式是：**在工作流程中强制更新**。

add-feature 和 fix-bug 的 skill 中都内置了"完成后运行 update-docs"的步骤。代码变更后，文档同步更新——不依赖个人自觉，而是通过流程保证。同时，由于 AI 每次都会读取这些文档，文档不准确会直接导致错误代码的产生，维护动力天然强于传统文档。

## 五、五个标准工作流

模板内置了 5 个 skill，覆盖日常开发的核心场景：

```
                   ┌────────────────┐
                   │  understand-   │
                   │    project     │  ← 首次接触，建立全景认知
                   └───────┬────────┘
                           │
              ┌────────────┼────────────┐
              ▼                         ▼
      ┌──────────────┐         ┌──────────────┐
      │  add-feature │         │   fix-bug    │  ← 日常开发
      └──────┬───────┘         └──────┬───────┘
             │                        │
             └───────────┬────────────┘
                         ▼
               ┌──────────────┐
               │  review-code │  ← 变更审查
               └──────┬───────┘
                       ▼
               ┌──────────────┐
               │  update-docs │  ← 同步文档
               └──────────────┘
```

每个 skill 的定位：

- **understand-project**：读取架构文档和模块文档，输出项目全景概要。新成员（或新的对话上下文）的第一步。
- **add-feature**：从需求分析到代码实现，内置"先读文档 → 设计方案 → 编写代码 → 运行测试 → 更新文档"的完整流程。
- **fix-bug**：强制"先复现 → 定位根因 → 再修复"的流程，避免 AI 看到报错就盲目修改。
- **review-code**：对照模块文档审查变更，重点检查接口契约是否被破坏、依赖方向是否违反。
- **update-docs**：代码变更后同步更新相关文档，确保文档不腐化。

这五个 skill 形成闭环：**理解 → 开发 → 审查 → 更新文档 → 下次开发时文档仍然准确**。

## 六、效果对比

使用与不使用这套模板，差异如下：

| 维度 | 裸用 AI 编码助手 | 使用本模板 |
|------|-----------------|-----------|
| 项目理解 | 每次对话从零开始推断 | 常驻约束 + 按需加载文档 |
| 代码一致性 | 依赖 AI 自行推断风格 | 约定明确写在文档中 |
| 跨模块安全 | 无法感知改动的影响范围 | 模块依赖和数据流有据可查 |
| 工作流程 | AI 自由发挥，输出质量不稳定 | skill 固化流程，步骤可控 |
| 文档维护 | 文档写完逐渐过时 | 开发流程内置文档更新 |
| Token 效率 | 要么信息不足，要么过度加载 | 渐进式披露，按需供给 |

简而言之：**裸用相当于让 AI 做没有上下文的临时工，用模板则是让 AI 成为有项目背景的团队成员。**

## 七、快速上手

将模板文件复制到项目中：

```bash
cp CLAUDE.md docs/ .claude/ your-project/
```

然后根据项目阶段选择对应的路径。

### 路径 A：已有代码的项目

项目已有代码但缺少结构化文档，让 AI 读代码自动生成：

**第一步：最小起步（2 分钟）**

1. 手动填写 `CLAUDE.md` 中的技术栈、命令等基本信息
2. 在项目目录下启动 AI 编码助手，输入：

```
请阅读项目代码，然后按照 docs/ 下的模板格式，
自动填充 architecture.md、模块文档和 data-flows.md。
```

AI 读取代码生成初稿，开发者快速 review 修正即可。相比从零手写，效率可以提升一个数量级。

**第二步：渐进扩展（按需）**

项目复杂度增长后，再逐步补充：

- **模块文档**：给核心模块创建 `docs/modules/[模块名].md`，基于 `_module-template.md` 填写接口契约
- **数据流文档**：在 `docs/data-flows.md` 记录核心业务的跨模块调用链
- **标准工作流**：把 `.claude/skills/` 复制到项目中，团队统一使用

### 路径 B：全新项目（尚未写代码）

这是更理想的使用方式——**文档先行，代码后行**。AI 从第一行代码开始就拥有完整的项目上下文。

你只需要能回答三个问题：
- **做什么**：项目目标和核心功能
- **用什么**：技术栈选型
- **怎么分**：模块划分和依赖方向

**第一步：填写核心文档**

手动填写两个文件：
1. `CLAUDE.md` — 技术栈、包管理器、代码风格等约束
2. `docs/architecture.md` — 模块列表、依赖方向、通信模式

不需要写得很详细，先把大方向定下来。例如：技术栈选 TypeScript + React + Express，分为 auth、order、inventory 三个模块，auth 不依赖其他模块，order 依赖 auth 和 inventory。

**第二步：让 AI 基于文档初始化项目**

```
请阅读 CLAUDE.md 和 docs/architecture.md，
按照架构设计初始化项目目录结构和基础代码框架。
```

AI 会根据文档中定义的技术栈、模块划分和依赖方向，生成项目脚手架。因为有完整的上下文，生成的代码从一开始就符合架构约定。

**第三步：迭代开发**

基础框架就绪后，使用 add-feature skill 逐步添加功能。每次开发 AI 都会先读文档再动手，保持代码与架构的一致性。随着功能增加，通过 update-docs skill 同步更新模块文档和数据流文档。

## 八、FAQ

**Q：我的项目很小 / 就几个文件，需要这些吗？**

小项目用 CLAUDE.md 一个文件即可，写明技术栈和基本约束。等项目规模增长后再添加 architecture.md 和模块文档。这套模板本身就是渐进式的，无需一步到位。

**Q：只支持 Claude Code 吗？其他 AI 工具能用吗？**

核心内容是纯 Markdown，任何 AI 工具都能使用。`CLAUDE.md` 是 Claude Code 的约定（自动加载），Cursor 对应 `.cursorrules`，其他工具也有类似机制。`docs/` 和 `skills/` 的内容对所有 LLM 通用——将文档内容提供给 AI 即可。

**Q：文档会不会很快过时？**

传统文档过时的原因是没人读、缺乏维护动力。这套文档有本质区别——AI 每次都会读取，文档不准确会直接导致错误代码的产生。加上 update-docs skill 在流程中强制同步，过时概率比传统文档低得多。当然，再好的流程也需要人工 review，但至少从"完全无人维护"变成了"有流程保障"。

**Q：可以自定义 skill 吗？**

完全可以。每个 skill 就是一个 Markdown 文件，可以根据团队需求自由增删改。例如添加 `write-test` 专门生成测试用例、添加 `deploy` 定义发布流程、添加 `migrate-db` 处理数据库迁移——格式一致，复制后修改即可。

## 最后

这套模板的核心理念很简单：**AI 编码助手并非能力不足，而是缺少上下文。提供一份结构化的项目心智模型，它就能像一个熟悉项目的团队成员一样工作。**

最小成本的尝试方式：复制 CLAUDE.md，填入项目信息，体验一下差异。
